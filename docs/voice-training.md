# Voice Training

## Commands
- `/voice train <wiki> "<character>"` — scrape wiki, generate structured persona card, greetings, offline quips, custom mood names, structure hints via 5 parallel Ollama calls, then run ASCII art generation serially afterward (the art prompt consumes the fresh persona).
- `/voice regenerate [name|--all]` — re-run *every* LLM-backed generator for an existing profile: persona, anchor_lines, banned_names, greetings, offline_quips, moods, structure_hints, ASCII art, and mood_frames. Preserves `lines`, finetune metadata, `source`, and `created`. ~60s per voice on Qwen3-14B. Omit the name to regenerate the active voice.
- `/voice ascii [name|--all]` — re-run *only* the ASCII art generator plus mood_frames render. Use when the text data is good but the frames came out off-model. Much faster than `regenerate`. Hot-swaps the refreshed frames + mood sets into the running `PersonalityEngine` if the active voice was the target.

Every regeneration now emits a `Classified: skeleton=... (hair=#... skin=#... outfit=#... accent=#... shadow=#... highlight=#...) zones={...}` line into the chat log immediately after the classifier resolves, so the JSON behind each render is visible without tailing the log file.

## When to use `ascii` vs `regenerate`
- Wrong proportions, misaligned mouth frames, broken Rich-markup color, mirrored face, wrong zone pick (e.g. missing beard): `ascii`
- Voice feels off-character, moods wrong, catchphrases stale: `regenerate`
- Model drifted so hard the underlying lines feel wrong: `/voice train` again

`ascii` is safe to re-run repeatedly. Each call rerolls the classifier; the fallback ladder is `cloud Haiku → local Qwen3 → _DEFAULT_CLASSIFICATION` so you'll always have something to look at, and you can keep hitting `/voice ascii` until a roll lands.

## Persona Format
Structured VOICE/CATCHPHRASES/NEVER/WORLDVIEW/VISUAL card using "You [verb] [pattern]" instructions, not personality adjectives. Generated by `_generate_persona()` in `train_voice.py` with 25 sample lines, max_tokens=500.

Example:
```
VOICE: You speak in short, excited bursts. You use math words as exclamations.
CATCHPHRASES: "Mathematical!", "Algebraic!", "What the cabbage?!"
NEVER: Modern internet slang ("dumpster fire", "lowkey"). Cynicism or sarcasm.
WORLDVIEW: Everything is an adventure. Helping people is what heroes do.
VISUAL: White bear-ear hood, pale peach skin, cyan tee, navy shorts, green backpack strap.
```

The VISUAL line is generated by a separate `_generate_visual_tells()` call and spliced into the persona via `attach_visual_tells`. It explicitly prompts the model to surface zone-relevant details (beard shape, unusual eyes, chest motifs, tail / trailing element) so the downstream classifier has something to anchor on. `UNKNOWN` is a valid response — the expanded rubric is careful not to pressure the model into hallucinating a beard or tail that the character doesn't actually have.

## Anchor Lines
Top 150 most distinctive dialogue lines scored at training time via `_score_line()` heuristic (length sweet spot, catchphrase presence, personality markers). Used for tiered few-shot sampling at runtime: 60% from anchor pool, 40% from general pool, 2 anchors placed last for recency effect.

## Cross-Franchise Guardrails
- `banned_names`: characters from OTHER franchises, auto-derived from source URL via `_derive_banned_names()`
- Response filter in `PersonalityEngine` suppresses any output mentioning a banned name (e.g. Bender mentioning Finn)
- Franchise mapping in `_FRANCHISE_CHARACTERS` dict covers Adventure Time, Futurama, Regular Show

## ASCII Art — Skeletons + Zones

### Pipeline
Generation lives in `_generate_ascii_art` (`tokenpal/tools/train_voice.py`):

1. **Classify + palette (LLM)** — one call to the cloud Haiku classifier (if `cloud_llm.enabled AND cloud_llm.voice_classifier`) or local Qwen3 otherwise. Returns JSON with a skeleton name, 6-color hex palette (`hair`, `skin`, `outfit`, `accent`, `shadow`, `highlight`), single-glyph eye + mouth picks, and a 5-field zones map. Cloud path uses the same prompt as local but gets materially better canonical-character recall.
2. **Validate (`_parse_classification_json`)** — extracts the outermost `{...}` blob from prose, checks the skeleton name against `SKELETONS`, validates each palette entry as `#RRGGBB`, falls back `highlight → outfit` when the model omits it, rejects multi-codepoint or bracket-containing glyphs, and routes `zones` through `normalize_zones` so any illegal pick gets coerced to `"none"` for the target skeleton.
3. **Fallback** — if both paths return None, `_DEFAULT_CLASSIFICATION` renders a neutral grey humanoid. Never crashes.
4. **Render (`_render_skeleton_frames`)** — substitutes the palette + glyphs + zones into the chosen template three times to produce idle (default glyphs), idle_alt (eye → `─` blink), and talking (mouth swapped via `_TALKING_MOUTH` lookup, e.g. `▽ → ◇`). Returns `(idle, idle_alt, talking, classification)` so callers can re-render with mood overrides later.
5. **Mood frames (`_render_mood_frames`)** — walks `profile.mood_roles` and, for every canonical role in `_MOOD_EYE_OVERRIDES` (sleepy / bored / hyper / impressed / concerned), renders an idle/idle_alt/talking triple with the mood-specific eye glyph swapped in. Default mood is intentionally absent — the runtime falls back to `ascii_idle` when the active role isn't a key.
6. **Load-time markup heal (`ascii_renderer._fix_markup`)** — junk-prefixed hex (`[#$ff6600]` → `[#ff6600]`), Rich-only color names (`silver` → `#c0c0c0`), stray HTML tags, and any line that still fails `Text.from_markup` gets every bracket tag stripped as a last resort.
7. **Runtime crash guard (`BuddyWidget._render_frame`)** — catches `MarkupError` at frame render time and falls back to escaped plain text, so a surviving malformed profile can never bring down the UI.

Profile writes back via `save_profile` (non-atomic `write_text`). Single-voice runs finish in ~5-8s. Don't run `/voice ascii --all` over a flaky SSH connection.

### Skeleton templates (10 total)
`tokenpal/ui/ascii_skeletons.py` — hand-drawn 14-row Rich-markup templates with format slots for palette + face glyphs. Preview: `python -m tokenpal.ui.ascii_skeletons`.

| Skeleton | Canonical examples |
|---|---|
| `humanoid_tall` | Finn, Mordecai, Marco, generic protagonist |
| `humanoid_stocky` | Dexter, Muscle Man, compact bros |
| `robot_boxy` | BMO, Bender, tin-can robots |
| `creature_small` | Nibbler, chibi pets |
| `mystical_cloaked` | Ice King, wizards, hooded sorcerers |
| `ghost_floating` | Ghosts, spirits, floating orbs |
| `animal_quadruped` | Jake (dog form), four-legged pets |
| `winged` | Angels, Prismo-adjacent, fairies |
| `blob_amorphous` | Lumpy Space Princess, Peppermint Butler, talking food |
| `hand_creature` | Hi Five Ghost, Thing, Rayman-style disembodied hand |

Each skeleton uses `{hair}` / `{skin}` / `{outfit}` / `{accent}` / `{shadow}` / `{highlight}` / `{c}` / `{eye}` / `{mouth}` format slots. `_pad_line` centers every rendered line to `CELL_WIDTH=29`.

### Palette slot rules
The classifier prompt carries a worked Ice King example that pins each hex slot to prevent slot-swapping:
- **`hair`** — hair AND beard color only. Never skin, never hat, never crown.
- **`skin`** — face tone only.
- **`outfit`** — primary clothing (robe / shirt / armor).
- **`accent`** — trim and accessories: buttons, gems, crowns, belts, weapon highlights.
- **`shadow`** — always a darker hue of `outfit`, never a contrasting hue.
- **`highlight`** — one shade brighter than `outfit`, used by zone overlays for sheen / crown gleam / wing tip.

If a character has white hair with blue skin (Ice King, Meta Knight), the prompt enforces that these MUST be different hex values so the hair/beard renders visibly against the skin.

### Zones
Zones are addressable regions that the classifier fills in after picking a skeleton, letting one template serve many characters. Source: `tokenpal/ui/ascii_zones.py`.

| Zone | Mode | Options | Supported skeletons |
|---|---|---|---|
| `headwear` | prepend | none, crown, hood_with_ears, antenna, halo, wizard_hat, spikes | all 10 (subsets vary) |
| `facial_hair` | replace | none, beard_long, beard_stubble, beard_goatee, mustache_thick, beard_wide | humanoid_tall, mystical_cloaked |
| `body_motif` | replace | none, screen_dpad, chest_door | robot_boxy |
| `eye_region` | replace | none, single_cyclops, oversized_spiral | humanoid_tall, animal_quadruped |
| `trailing` | append | none, tail_curly, hair_drift | animal_quadruped, ghost_floating |

Three zone modes:
- **prepend** — overlay rows added above the skeleton template. `headwear` sits here.
- **replace** — overlay rows overwrite a fixed row range on the rendered body, declared in the zone's `targets` map. `facial_hair` / `body_motif` / `eye_region` sit here.
- **append** — overlay rows are added below the body. `trailing` sits here.

Each zone is a single `ZoneSpec` dataclass bundling `overlays`, `rubric`, `compat`, and (for replace-mode) `targets`. Registries `_ZONE_MODES`, `_REPLACE_OVERLAYS`, `_APPEND_OVERLAYS`, `_REPLACE_TARGETS`, and `_ZONE_COMPAT` auto-derive from a single `_ZONES` list, so adding a sixth zone is a one-`ZoneSpec` edit plus an append — no more five-parallel-dict-edits pattern.

Legacy module-level constants (`HEADWEAR_OVERLAYS`, `FACIAL_HAIR_RUBRIC`, etc.) are preserved as aliases pointing into the spec so existing imports in tests and train_voice keep working.

### Adding a new zone
1. Draft a `ZoneSpec(name=..., mode=..., overlays=..., rubric=..., compat=..., targets=...)` in `ascii_zones.py`.
2. Append it to `_ZONES`. The derived registries pick it up automatically.
3. Splice `SPEC.rubric_block_for_prompt()` (or the legacy `rubric_block(SPEC_RUBRIC)` alias) into `_build_classifier_prompt` with a one-line intro.
4. Add tests in `tests/test_ascii_zones.py` covering: legal rendering, illegal coercion, and row-range behavior for replace-mode overlays.

### Adding a new skeleton
1. Draw a 14-row Rich-markup template and add it to `SKELETONS`.
2. Add a sample palette entry to `_SAMPLE_PALETTES` (remember `highlight`).
3. Add `"new_skeleton": {"none", ...}` to EVERY zone's compat set in `ascii_zones.py`. Replace-mode and append-mode overlays need no new entries for `"none"` — the short-circuit in `apply_replace_zones` / `trailing_suffix` keeps missing skeleton keys safe.
4. Add a line to the classifier prompt's template list in `_build_classifier_prompt`.
5. Extend test coverage loops in `tests/test_ascii_zones.py`.

### Why skeleton templates + classification instead of freeform LLM art
Small open-weight LLMs (gemma4, Qwen3-14B) can't reliably draw ASCII characters. The skeleton library replaces "draw a 14×30 character" with "classify into 1-of-10 + pick 6 hex colors + pick 2 glyphs + pick 5 zone options" — a task LLMs are genuinely good at. Hand-drawn skeletons give a deterministic quality floor; variety comes from palette + zone combos (thousands of unique renders across the 10 skeletons).

## Mood-aware frames

Per-mood frame triples let the same character render differently depending on the active mood. Bender's eyes turn to slits (`─`) when he's sleepy; Rick goes wide-eyed (`◉`) when impressed.

### How it works
- **Train time**: `_render_mood_frames(classification, mood_roles)` walks every role in the profile's `mood_roles` dict, looks up a heuristic eye override in `_MOOD_EYE_OVERRIDES` (sleepy → `─`, bored → `○`, hyper → `◉`, impressed → `◉`, concerned → `◎`), and renders a full idle/idle_alt/talking triple with the overridden glyph. Output is stashed on `profile.mood_frames` keyed by canonical role name (lowercased).
- **Runtime load**: `_load_voice_art` in `app.py` builds `BuddyFrame.mood_frame_sets(personality.voice_mood_frames)` alongside the default triple and calls `overlay.load_voice_frames(frames, mood_frame_sets)`. The overlay stashes both.
- **Mood transitions**: the brain orchestrator's `_push_mood_if_changed` helper fires `overlay.set_mood(role)` only when `personality.mood_role` transitions. The role key (stable across voice swaps — sleepy/bored/hyper/impressed/concerned/default) is what the overlay looks up, NOT the custom display name (LSP's "LUMPY" doesn't change the role, just the label).
- **Fallback**: if the current mood isn't in `mood_frames`, `BuddyWidget._get_frame` falls back to the default `_custom_frames`, then to the built-in `BuddyFrame.get(name)`. Legacy profiles without `mood_frames` render exactly as before.

### Default is intentionally absent from mood_frames
The default mood reuses `profile.ascii_idle` / `ascii_idle_alt` / `ascii_talking` so the default render and the mood_frames rendering share one surface. Only moods that actually change the glyph get a stored triple.

### Upgrading to LLM-picked mood glyphs
Heuristic overrides cover the canonical six-role table. A character with canon-specific mood expressions (grumpy Bender's slit eyes that aren't the generic sleepy `─`) would benefit from a dedicated LLM call at train time. Plan lives in the phase 3 retro; implement by adding a `_generate_mood_glyphs` function that takes character + mood_roles and returns `dict[role, {eye, mouth}]`, then thread those picks through `_render_mood_frames`.

## Voice Priming
The voice reminder at the generation point uses actual catchphrases instead of meta-instructions:
- Old: `(Remember: you are Finn. Stay in character. No emojis.)`
- New: `(Finn's style: "Mathematical!", "Algebraic!", "Shmowzow!")`

This exploits the recency effect — the model's next-token prediction is primed by character-specific tokens.

## Shared Utilities
`voice_profile.py` exports: `franchise_from_source()`, `parse_catchphrases()`, `parse_visual_tells()`, `attach_visual_tells()`, `FANDOM_NAMES` — used by both training pipeline and runtime.

## Profile Storage
- Profiles saved to `~/.tokenpal/voices/<slug>.json`, auto-activated in config.toml
- Voice persona replaces default TokenPal identity (not appended)
- Custom moods: pipe-delimited prompt, `_parse_custom_moods()` regex parser
- `mood_frames` field: empty dict for legacy profiles, populated dict for post-phase-3d profiles. `load_profile` defaults the field to `{}` on absence — no migration script required.
- Version field: v1 = old one-sentence persona, v2 = structured card

## Cloud Haiku classifier (opt-in)
Haiku's canonical-character recall is materially better than Qwen3-14B for iconic characters (Ice King, Bender, LSP). Opt in with:

```
/cloud anthropic enable <key>
```

Then flip the voice classifier flag — either via the Voice modal (F4) checkbox or by setting `[cloud_llm] voice_classifier = true` in config.toml.

The classifier prompt is identical across cloud and local paths. The diagnostic line in the chat log prefixes the classification with its source (`cloud` / `local` / `default_fallback`) so silent fallbacks are visible.
