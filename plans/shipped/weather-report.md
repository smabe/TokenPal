# weather-report

## Goal
Port the full weather-environment visualization to the Qt overlay: scaled-down sun/moon in the screen's top-right corner, overcast double-cloud stack, drifting cloud with rain/snow, lightning flashes during storms, rain/snow visually interacting with the buddy (drops land on his head, splash off, snow dusts his shoulders), and an occasional shooting star bisecting the moon at night. Fed by the existing `EnvironmentSnapshot` provider already wired into `QtOverlay` (no brain changes).

## Non-goals
- No changes to Textual overlay (`tokenpal/ui/textual_overlay.py`, `ui/ascii_props.py`, `ui/buddy_environment.py`) â€” Qt gets its own rendering path, shared data model only.
- No full-sky ambient starfield â€” explicitly replaced with rare shooting stars. No dust motes during clear days.
- No hot-outside sweat-bead steam glued to the Qt buddy â€” park until we decide how reactions attach to a swinging pendulum.
- No rain/snow physical coupling with the pendulum â€” drops land on the buddy visually (stop, splash, fade) but do NOT push him or impart torque. The `PendulumSimulator` state stays isolated.
- No puddle on the dock, no wet speech bubble, no umbrella prop.
- No new `Kind` values or WMO remapping. Reuse `wmo_to_kind` / `EnvState.from_inputs`.
- No new sense and no config toggle â€” Qt weather is always on whenever `[senses] weather` is feeding data. Disabling is done upstream at the sense, not at the overlay.
- No porting of `ParticleField`'s full particle zoo (no dust, no steam, no dizzy swirl, no impact burst) â€” the Qt weather widget implements only the kinds listed in Goal.

## Failure modes to anticipate
- **Multi-monitor / screen change**: `QScreen.availableGeometry()` on primary is the simple answer, but users move monitors. Listen for `QGuiApplication.primaryScreenChanged` or recompute top-right on every `screenChanged` signal; otherwise the widget strands itself off-screen.
- **HiDPI scale**: font metrics in `BuddyWindow` already do a paint-measurement trick (`_measure_block_paint_width`) to work around Qt's width lies on retina. The weather widget must reuse the same approach or the sprite columns will jitter vs. ParticleField coordinates.
- **Input passthrough**: the widget must NOT eat clicks. On macOS frameless transparent windows, `WA_TransparentForMouseEvents` is necessary; on Wayland `setWindowFlags(Qt.WindowTransparentForInput)` plus the attribute. Test both or users lose menu-bar / desktop clicks under the widget.
- **Always-on-top + accessory mode**: BuddyWindow uses `apply_macos_stay_visible`. WeatherWindow needs the same treatment or it'll slip behind full-screen apps and re-appear mid-drag. Call after `show()`.
- **Translucent paint**: per `CLAUDE.md` "UI / Qt Conventions" â€” stylesheets can be silently blocked by `WA_TranslucentBackground`. Paint sprites in `paintEvent`, not via stylesheet. Clear the frame each tick with `painter.fillRect(rect, Qt.transparent)` inside a composition-mode-source block or the stale frames ghost.
- **Sensitive-app clear (intentional divergence from Textual)**: Textual freezes CloudDrift and ParticleField mid-state (both early-return on `sensitive_suppressed`). Qt version intentionally *clears* all in-flight particles + shooting stars instead of freezing â€” a paused rain drop mid-air is more attention-drawing than no weather at all, and the whole point of sensitive-suppression is to go invisible. Label this in a code comment so a later reader doesn't "fix" it back to the Textual behavior.
- **Env provider called on the wrong thread**: `_env_provider` is `brain.environment_snapshot`, which reads a cached dict. It's cheap and read-only, but the brain might update it while Qt reads. Snapshot to a local var at tick start; don't iterate the mapping twice.
- **Day/night boundary flicker**: `EnvState.from_inputs` derives `is_day` from `datetime.now().hour`. At 19:00 sharp the sun will instant-swap to moon. Acceptable (same behavior as Textual), but call it out so a reviewer doesn't file a bug.
- **Shooting star placement**: we want "bisected moon" â€” the star's path has to actually cross the moon's bounding box, not random sky. Compute the path as a line through the moon center with a Â±20Â° jitter, clip to widget rect. Schedule via a Poisson process at ~1 every 45â€“90 s so it feels rare.
- **Rain drift artifacts**: the cloud drifts `Â±4` cells on a 45 s cosine (shared `CloudDrift` period). If rain spawns at the cloud's instantaneous X but drops with constant vertical velocity, drops *line up* as parallel diagonals â€” looks like a machine, not weather. Fix by jittering spawn offset inside the cloud's width and adding a small wind-vx term shared with the drift derivative.
- **Sprite color caching**: Textual path pre-warms `Style` objects in `ascii_props.py`. Qt has no equivalent cache; building `QColor` per glyph per tick is fine at 60 Hz Ă— ~30 sprite cells Ă— 60 particles, but benchmark before assuming. Cache `QColor` keyed on hex if profiling flags it.
- **Teardown order**: if the weather timer outlives the widget, we get use-after-free. Stop the timer in `closeEvent` AND have `QtOverlay.teardown()` deleteLater it explicitly.
- **Sky â†” buddy-overlay seam**: two windows render one physical particle. Rain falls in the sky window, crosses an invisible handoff Y, then renders in the buddy-rain overlay until it lands / despawns. If each widget runs its own sim you get tearing. Fix: `WeatherSim` owns all particles; each widget renders only the subset with y inside its world-rect on paint, no per-widget state. Test a drop crossing the seam â€” no visual gap, no double-draw.
- **Buddy rotation on rain contact**: `PendulumSimulator` rotates the buddy's art. The naive world-space AABB grows huge at 45Â° and eats rain that visually shouldn't hit him. Approach (pinned): invert `BuddyWindow._build_transform()`, map each falling drop's screen position into art-frame, hit-test against the tight untransformed art AABB there. Do NOT mix with world-space OBB checks â€” one approach only.
- **Lightning flash (new design, not a port)**: Textual lightning is a single 4-segment bolt with 0.25 s life and no alpha strobe. Qt version is a new design: strobe the sky window's alpha. Cap max-alpha (0.45), cap frequency (â‰¤ 1 flash per 8 s), cap duration (â‰¤ 80 ms on + 40 ms off + 40 ms on) and tie to `intensity` only for stormy kinds. Do NOT flash the buddy-rain overlay (doubled flash).
- **Snow accumulation**: Textual version doesn't accumulate â€” snow falls past, that's it. Accumulating snow on the buddy's shoulders means tracking per-column dust on a rotating widget and re-projecting through `_build_transform`. Keep it dumb: a thin 1-cell "dust" line above the buddy's top-art-row, alpha-weighted by recent snow-drop density over the last N seconds, cleared on drag. No persistence across sessions.
- **Overcast double-stack anchoring**: Textual uses `anchor_dx` / `anchor_dy` offsets on a shared 14-cell sprite width, both sprites tied to one `CloudDrift` clock at anti-phase Ď€. Port the data (`ascii_props._OVERCAST_CLOUD_LINES` + the two offsets) verbatim so the visual matches; resist the urge to redraw â€” the drift math is load-bearing.
- **Scope creep surface**: with lightning + snow + overcast + buddy-contact in one PR, the failure count in `tests/test_qt_weather.py` can easily hit 20+ cases. Prioritize: buddy-contact catches, sensitive-suppressed clears, intensity monotonicity. Leave visual-correctness to manual smoke.
- **`position_changed` fires every physics tick**: not just on movement â€” the timer emits it unconditionally while active (buddy_window.py:344 comment). `BuddyRainOverlay._reanchor()` must be O(1) and idempotent: early-return if the cached rect hasn't changed beyond 1 px.
- **macOS hidden-vs-visible footgun in tests**: translucent frameless windows auto-hide between event pumps. All tests must assert the user-intent flag (`.isHidden()` / widget-level state), not `.isVisible()`. Existing `test_qt_overlay.py:155` is the template.

## Done criteria
- Clear day: small sun in screen's top-right corner. Clear night: small moon, zero ambient starfield, roughly one shooting star per 45â€“90 s streaking *through* the moon's bounding box (not random sky).
- Partly cloudy: luminary + one drifting cloud, no rain. Overcast (intensity â‰Ą 0.7): luminary + the double-cloud stack at anti-phase drift.
- Rain: cloud drifts calmly, droplets fall from its footprint, intensity scales spawn rate (drizzle < rain < storm). Storm adds a dim, bounded lightning flash on the sky window.
- Snow: snow sprite drifts down slower than rain with lateral sway, intensity scales spawn rate.
- Buddy interaction: rain/snow particles that fall onto the buddy's rotated art rect stop on contact. Rain spawns a brief splash (2-3 glyph radial), snow leaves a short-lived shoulder-dust line. Drops NEVER push or torque the pendulum. Dragging the buddy clears any accumulated snow dust.
- Sky window and buddy-rain overlay are both input-transparent â€” clicks through to the desktop.
- `sensitive_suppressed = True` instantly clears all sprites and particles from both windows; clearing the flag resumes cleanly from empty state.
- Hiding the buddy hides both weather windows; showing the buddy restores them. Tray Quit stops the timer and tears down both widgets cleanly (no "QTimer used outside its thread" warnings).
- `tests/test_qt_weather.py` green under `pytest`. Full suite green. `ruff check tokenpal/` green. `mypy tokenpal/ --ignore-missing-imports` green.
- Manual smoke on macOS with `./run.sh` at each of: clear day, clear night, overcast day, rain, storm, snow (force by editing the cached `weather_data` or `weather_code` in config/dev helper). Drag the buddy through the rain and confirm drops land on his head regardless of rotation angle.
